# Habit Tracker

사용자 인증 후 **습관을 생성하고 관리**하며,  
각 습관의 **하루 기록을 체크하고 흐름을 확인할 수 있는 프론트엔드 프로젝트**입니다.

단순 CRUD 구현에 그치지 않고,  
**인증 흐름**, **API 에러 표준화**, **서버 상태 관리**, **커스텀 훅 분리**,  
**CSV 내보내기**, **차트 시각화**, **디자인 토큰 기반 테마 전환**까지 포함해  
프론트엔드 구조를 고민하며 구현했습니다.

## 주요 기능

### 1. 인증

- 로그인 / 회원가입
- 인증된 사용자만 접근 가능한 라우트 구성
- 인증 만료 시 전역에서 감지 후 토큰 제거 및 만료 상태 처리

### 2. 습관 관리

- 습관 생성 / 조회 / 수정 / 삭제
- 습관별 상세 페이지 분리
- 모달 기반 생성 / 수정 플로우

### 3. 하루 기록 관리

- 날짜별 기록 저장 / 수정 / 삭제
- 완료 여부, 장소, 난이도, 컨디션, 메모 관리
- 기록 흐름에 맞춘 리스트 UI 구성

### 4. 시각화 및 내보내기

- 차트를 통한 기록 통계 확인
- CSV 내보내기 기능 제공
- 엑셀 한글 깨짐 방지를 위한 BOM 처리

### 5. 사용자 경험

- 토스트 알림 인터페이스 분리
- 디자인 토큰 기반 테마 전환
- 반응형 UI 적용
- 404 / 403 / 401 등 예외 상황 분기 처리

## 구현하면서 중요하게 본 점

### 1. 인증 만료 처리의 중앙화

페이지마다 401을 분기하는 방식은 중복이 많고 유지보수가 어려웠습니다.  
이를 줄이기 위해 **React Query 전역 에러 처리 + 라우터 가드** 구조로 정리했습니다.

- Query / Mutation 에러에서 인증 만료를 감지
- 토큰 제거 및 만료 상태 저장
- 라우터 레벨에서 접근 제어

이를 통해 페이지별 인증 분기 부담을 줄이고,  
인증 흐름을 한 곳에서 관리할 수 있도록 구성했습니다.

---

### 2. API 에러 표준화

초기에는 HTTP 에러를 호출부에서 각각 다르게 해석할 여지가 있었습니다.  
이를 줄이기 위해 `ApiError`를 두고 에러를 코드 단위로 표준화했습니다.

예시:

- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `NETWORK_ERROR`
- `TIMEOUT`

이 구조를 통해 호출부에서 **일관된 기준으로 예외를 분기**할 수 있도록 했습니다.

---

### 3. 서버 상태와 UI 상태의 분리

습관 데이터와 하루 기록은 **서버 상태**로 보고 React Query로 관리하고,  
모달 열림 여부나 토스트 등은 **UI 상태**로 분리했습니다.

이를 통해:

- 서버 데이터 캐싱 / 무효화 기준을 명확히 하고
- UI 상태가 데이터 흐름과 뒤섞이지 않도록 구성했습니다.

---

### 4. 커스텀 훅을 통한 책임 분리

컴포넌트 내부에 조회 / 액션 / 다운로드 로직이 섞이면  
화면 책임과 흐름 책임이 함께 커지는 문제가 있었습니다.

이를 줄이기 위해 다음과 같은 로직을 커스텀 훅 또는 별도 함수로 분리했습니다.

- 조회 로직
- 알림 인터페이스
- CSV export
- 공통 유틸 함수

이 과정에서 **“컴포넌트가 알아야 할 것만 남기고, 나머지는 밖으로 밀어내는 방식”**을 기준으로 삼았습니다.

---

### 5. 기능보다 설명 가능한 구조 지향

이 프로젝트에서는 단순히 기능을 붙이는 것보다  
**왜 이렇게 분리했는지 설명 가능한 구조**를 만드는 것을 중요하게 봤습니다.

그래서 작업 중에도 다음 질문을 계속 기준으로 삼았습니다.

- 이 로직은 어디에 있어야 하는가?
- 이 구조가 최선인가?
- 다른 방법은 없는가?
- 이 선택을 PR이나 면접에서 설명할 수 있는가?

## 아쉬웠던 점

### 1. 테스트 코드 부족

프로젝트를 진행하면서 구조 정리와 기능 구현에 우선순위를 두다 보니  
테스트 코드까지 충분히 확장하지는 못했습니다.

### 2. 기능 우선 구현으로 인한 후반 리팩토링

초반에는 기능 완성에 집중하면서,  
후반에 인증 흐름 / 에러 처리 / query key / 훅 분리 등을 다듬는 작업이 필요했습니다.

## 프로젝트를 통해 배운 점

이 프로젝트는 단순한 습관 관리 앱을 넘어서,  
프론트엔드에서 다음을 어떻게 다룰지 고민하게 만든 작업이었습니다.

- 인증 흐름을 어디서 관리할 것인가
- 에러를 어떤 기준으로 표준화할 것인가
- 서버 상태와 UI 상태를 어떻게 분리할 것인가
- 커스텀 훅을 어떤 기준으로 나눌 것인가
- “기능 구현”을 넘어 “설명 가능한 구조”를 어떻게 만들 것인가

특히 작업을 진행하면서  
코드 한 줄에도 이유를 묻고,  
선택지를 비교하고,  
불필요한 부분은 걷어내는 습관이 중요하다는 점을 배웠습니다.

## 회고

이 프로젝트는 취업용 프론트엔드 포트폴리오로 시작했지만,  
진행하면서 단순한 CRUD 구현보다 더 많은 고민을 남긴 프로젝트였습니다.

정형화된 구조를 따라가는 대신,  
직접 문제를 정의하고 구조를 선택해야 했기 때문에 더 어려웠지만,  
그 과정에서 **구조를 설명하는 힘**, **책임을 분리하는 감각**,  
**이유 있는 선택을 하려는 태도**를 더 많이 배우게 되었습니다.
